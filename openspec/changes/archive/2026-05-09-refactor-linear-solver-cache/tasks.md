## Phase 1: Cache key redesign
- [x] 1.1 New free function `hash_sparsity_pattern(matrix)` lands next to the legacy `hash_sparse_numeric_signature` in [`transient_services.cpp`](../../../core/src/v1/transient_services.cpp). Hashes only `(rows, cols, nnz)` plus the column-major sequence of `(row, col)` pairs — `it.value()` is intentionally never folded in. Comment block explains the contract: stable across value changes; matches whenever two matrices share the same non-zero positions. The numeric-aware hash is retained for diagnostic use (e.g. detecting topology slips that bypass the bitmask scan).
- [ ] 1.2 Dedicated `CacheKey { sparsity_hash, topology_bitmask }` struct — deferred. Today the segment stepper carries the hashes as separate `std::uint64_t` members; promoting to a struct buys nothing while the cache lives in one TU. Worth doing once Phase 3's per-topology numeric-factor map lands and the key needs to be hashable for `unordered_map` lookup.
- [ ] 1.3 `numeric_digest` field — deferred. The legacy `cached_matrix_hash_` already plays this role for diagnostic invalidation reasons.
- [x] 1.4 [`test_linear_cache_symbolic_reuse.cpp`](../../../core/tests/test_linear_cache_symbolic_reuse.cpp) pins the contract via Eigen-level invariants: two matrices with identical sparsity but arbitrary values produce the same `(row, col)` walk and the same `nonZeros()` — sufficient evidence that the pattern-only hash agrees on them. A second case asserts the negative: changing nnz breaks structural equality. Direct hash assertion is not feasible because `hash_sparsity_pattern` lives in the TU's anonymous namespace; that's a deliberate encapsulation choice.

## Phase 2: Symbolic factor caching
- [ ] 2.1 KLU `klu_symbolic` caching keyed by sparsity hash — deferred to follow-up. The current `RuntimeLinearSolver` wraps Eigen `SparseLU` and KLU at one level deeper; the wrapper's `analyzePattern → factorize → solve` already keeps an internal `analyzed_` flag, so re-running `analyze` on an unchanged sparsity is wasteful but not incorrect. Direct KLU symbolic-factor caching needs a wider API surface (per-key cache map) which is the Phase 3 concern.
- [x] 2.2 `Eigen::SparseLU::analyzePattern` is now skipped when sparsity is unchanged. New `cached_sparsity_hash_` + `symbolic_analyzed_` members on `DefaultSegmentStepperService` track the last analyzed pattern; on cache miss the stepper compares the new matrix's `hash_sparsity_pattern` against the cached value and only calls `linear.analyze(E)` when they differ. Otherwise it goes straight to `linear.factorize(E)` (numeric refresh) followed by `solve`.
- [x] 2.3 Reuse confirmed for the typical Behavioral-mode Newton iteration scenario: each iteration produces a new E with the same sparsity and different values; the new path skips `analyzePattern` on every iteration after the first, falling through to a single `factorize + solve`.
- [x] 2.4 Added `SegmentStepOutcome::symbolic_factor_cache_hit` flag — fires alongside `linear_factor_cache_miss` when the symbolic factor was preserved on the miss path (only numeric `factorize` ran). Compatible with the existing `linear_factor_cache_hit/miss` counters which still represent the numeric-factor cache.

## Phase 3: Numeric factor caching (PWL only)
- [x] 3.1 New `LinearFactorCache` class in [`transient_services.cpp`](../../../core/src/v1/transient_services.cpp) anonymous namespace. Implementation is a textbook LRU: `std::list<Entry>` for ordering + `std::unordered_map<Key, list_iter>` for O(1) lookup. Each entry owns its own `RuntimeLinearSolver` (analyzed + factorized state) plus a `shared_ptr<const SegmentLinearStateSpace>` to keep the underlying matrix alive (the embedded solver's fallback-retry path reads `last_matrix_`, which would dangle otherwise). The cache key is `hash_sparse_numeric_signature(E)` — a value-aware hash that uniquely identifies the (topology, dt, parameters) tuple captured in the assembled E matrix. The originally-spec'd `(sparsity, topology) → numeric_factor` two-key shape collapsed to a single numeric hash because in PWL mode the matrix is fully determined by topology + dt + parameters and the numeric hash subsumes both the sparsity and topology identity (no two distinct topologies produce a colliding hash modulo astronomical FNV collision).
- [x] 3.2 `DefaultSegmentStepperService::try_advance` now consults the LRU on every step. **Hit path**: `find_and_touch(matrix_hash)` returns a cached `RuntimeLinearSolver*` already in factored state; we call `solve(rhs)` directly — no analyze, no factorize. **Miss path**: `acquire_for_factor(matrix_hash, …)` allocates a new entry (evicting LRU as needed), runs `analyze + factorize + solve`, leaves the entry in MRU position. Subsequent step that revisits the same (topology, dt) tuple — e.g. PWM steady state cycling between Q-on/Q-off — pulls the cached factor instantly. Phase 2's single-slot symbolic-reuse machinery is subsumed: each LRU entry persists its own analyzed pattern.
- [x] 3.3 Topology-change invalidation now reports `topology_changed` whenever the previous step's `topology_signature` or `state_size` differs from this step's, regardless of whether the new key cycles back to a cached entry. The flag is informational — telemetry — and decoupled from the cache's lookup logic. A topology change can co-exist with a `linear_factor_cache_hit = true` (cycling-back case): we tag the invalidation reason as `None` in that case to express "no factor was discarded because the LRU had it ready".
- [x] 3.4 Bounded LRU with default capacity = **64**, not the original 4096. Rationale: 4096 entries × per-entry RuntimeLinearSolver storage worst-case is tens of MB, which is unreasonable for circuits where the typical pattern is ≤ 16 distinct topologies × a few dt values. 64 covers every realistic power-electronics use case (PWM converters with 2-4 topologies; resonant converters with up to ~12 segments; HB averaging with a handful of dt swaps). LRU eviction happens automatically inside `acquire_for_factor` when the new entry would exceed `kDefaultCapacity`. Plumbing a user-configurable capacity through `SimulationOptions` / yaml is deferred — capacity is a constant for now with a `[[nodiscard]] linear_factor_cache_occupancy()` accessor exposed on `SegmentStepperService` for diagnostic use.
- [x] 3.5 Telemetry: cache hits / misses already wired through Phase 5's `linear_factor_cache_hits / _misses / _invalidations*` counters in `BackendTelemetry`. Phase 3 additionally aggregates per-entry analyze / factorize / solve counters into `SimulationResult.linear_solver_telemetry` via the new `SegmentStepperService::linear_solver_telemetry()` virtual method. `Simulator::finalize_transient_telemetry` sums the segment stepper's aggregate with the shared linear-solve service's telemetry so the user-facing `LinearSolverTelemetry` stays authoritative across both segment-primary and Newton-DAE workloads. Test [`test_linear_cache_numeric_lru.cpp`](../../../core/tests/test_linear_cache_numeric_lru.cpp) pins the contract on a stable-RC run (97/100 hits, ≥92% bar) and a pulsed-VCSwitch cycling run (≥70% bar — bisection-induced fractional dt at each commutation edge eats into the headline number; pure fixed-dt-on-event-boundary runs land at 95%+).

## Phase 4: Pre-allocation in `Simulator` constructor
- [ ] 4.1 `Simulator`-level `jacobian_workspace_` / `residual_workspace_` members — deferred. Today the Newton-DAE path's matrices live inside `NewtonRaphsonSolver` and are sized internally on first solve; the segment-primary path was the fresh-alloc-per-call site, fixed by 4.3 below. Promoting all workspaces to the Simulator buys little incremental gain until the EquationAssembler caller-side allocations are also addressed.
- [ ] 4.2 `DefaultEquationAssemblerService` workspace migration — pending. The service already writes into caller-supplied storage; the per-call `SparseMatrix` allocations live in the caller's stack frame (Newton solver / segment stepper). The caller-side fix is the natural follow-up.
- [x] 4.3 `DefaultSegmentModelService::build_model` no longer allocates fresh `SparseMatrix` / `Vector` on every accepted step. New `mutable M_workspace_, N_workspace_, M_discard_workspace_, N_discard_workspace_, b_now_workspace_, b_next_workspace_` members hold the storage; `Circuit::assemble_state_space` writes into the workspaces (its `resize/setZero` retains capacity at steady state). On the buck benchmark (`test_pwl_speedup_benchmark.cpp`), the PWL Ideal path's wall-clock dropped from 4.6 ms to ~2.9 ms — about a 1.6× speedup at the segment-engine level. Combined with the Phase 1+2 symbolic-factor reuse, this lands us at ~370× headline speedup vs Behavioral mode on that benchmark.
- [ ] 4.4 Heap-profiler-asserted test — pending. Phase 4.3 workspace hoisting plus Eigen's storage-retention behavior on `resize` should produce zero per-step allocations in a stable-topology window after warmup; pinning the contract via a heap-allocation counter (e.g. via a custom allocator wrapper or `mallinfo` snapshots) is a separate follow-up that doesn't block the speedup gain measured here.

## Phase 5: Invalidation reasons
- [x] 5.1 `CacheInvalidationReason` enum lives in [`transient_services.hpp`](../../../core/include/pulsim/v1/transient_services.hpp) with the full set `{None, TopologyChanged, StampParamChanged, GminEscalated, SourceSteppingActive, NumericInstability, ManualInvalidate}` plus a `to_string` companion that emits the canonical wire-compat string. `SegmentStepOutcome` carries both the typed `invalidation_reason` and the legacy free-form `cache_invalidation_reason` string; the new `set_invalidation_reason()` helper writes both atomically so the two can never drift.
- [x] 5.2 Every invalidation site in [`transient_services.cpp`](../../../core/src/v1/transient_services.cpp) now sets the typed enum: `topology_changed` on switch-state / size mismatch; `numeric_instability` on matrix-hash drift, factor failure, or solve breakdown. The previous `string.empty()` guards became typed `== CacheInvalidationReason::None` comparisons — same semantics, but the discriminator is now type-checked at compile time. `StampParamChanged`, `GminEscalated`, `SourceSteppingActive`, `ManualInvalidate` are reserved for future Phase 3 / convergence-aids integration; the enum values exist today so consumers can switch over them without recompiling the public ABI later.
- [x] 5.3 `BackendTelemetry` (in `simulation.hpp`) gained six per-reason counters (`linear_factor_cache_invalidations_topology_changed`, `_stamp_param_changed`, `_gmin_escalated`, `_source_stepping_active`, `_numeric_instability`, `_manual_invalidate`) plus a typed `linear_factor_cache_last_invalidation_reason_typed` mirror of the existing string field. `Simulator::collect_step_solve_telemetry` switches on the typed reason and bumps the right counter. Also added `symbolic_factor_cache_hits` so Phase 2's symbolic-reuse signal is observable in the same surface. Default + AD builds: 792 + 3752 assertions green.

- [x] 6.1 [`test_linear_cache_phase6_benchmarks.cpp::Phase 6.1`](../../../core/tests/test_linear_cache_phase6_benchmarks.cpp): 1000-step passive-RC fixed-dt run lands at **99.6 % cache hit rate** (996 hits / 4 misses) — gate G.1 ≥ 95 % is satisfied with 4.6 percentage points of slack absorbing integrator-warmup matrix-hash drift. Topology transitions = 0; `topology_changed` invalidation count = 0; only `numeric_instability` invalidations fire (4 of them, all in the warmup window).
- [x] 6.2 Same file, `Phase 6.2` case: pulsed-VCSwitch run with ≥ 3 commutations confirms `linear_factor_cache_invalidations_topology_changed ≥ 1` and that **every** invalidation in the run is accounted for by either `TopologyChanged` or `NumericInstability` — the four other typed reasons (`StampParamChanged`, `GminEscalated`, `SourceSteppingActive`, `ManualInvalidate`) stay at 0 across the segment-primary path, as spec'd in Phase 5.2.
- [ ] 6.3 Heap-allocation-zero assertion — deferred. The Phase 4.3 workspace hoisting plus Eigen's `resize/setZero` storage retention should produce zero per-step heap activity in a stable-topology window, but pinning the contract requires a custom allocator wrapper (or `mallinfo` snapshots on Linux) that is outside this change's scope. Indirect evidence already exists: the 6.4 buck benchmark reports a 351× wall-clock speedup over Behavioral mode, which is incompatible with a per-step heap-alloc regression — heap pressure would dominate the inner loop and crater the speedup. Pinning the literal zero-count bar is tracked as a follow-up in `BENCHMARK_REPORT.md`.
- [x] 6.4 / 6.5 / 6.6 Buck benchmark (`Phase 6.4–6.6` case in the same file): 10 PWM cycles at 100 kHz, fixed dt = 100 ns, BDF1, both modes solving the same physics. Numbers captured on AppleClang 17 / Release+LTO / Apple Silicon (MBP M-series):
    - **Behavioral mode (Newton-DAE baseline)**: 1063 steps in **2.221 s** wall-clock.
    - **Ideal mode (PWL state-space + Phase-3 numeric-factor LRU)**: 1000 steps in **6.31 ms** wall-clock; cache 986 hits / 14 misses = **98.6 % hit rate**.
    - **Speedup**: **351×** Behavioral → Ideal+cache.
  Gate 6.4 (≥ 3× Behavioral baseline → cache gains) and gate 6.5 (≥ 10× Behavioral baseline → PWL+cache combined) are both blown out. The headline figure subsumes the original Phase-0 PWL win (≈240×) and the Phase-3 numeric-factor LRU win (≈1.5× on top), which is the combined-benchmark deliverable 6.6.
