## Phase 1: Operating-point linearization
- [x] 1.1 New public struct [`LinearSystem`](../../../core/include/pulsim/v1/simulation.hpp) (`E, A, B, C, D, state_size, input_size, output_size, t_linearization, x_linearization, method, failure_reason`) plus `Simulator::linearize_around(x_op, t_op)` method. Mapping: `E = M`, `A = -N`, `B = b(t_op)` as a single column (Phase-4 multi-input matrix is the next refinement), `C = I_n` (full-state output; per-node selection follows in AC sweep), `D = 0`. PWL-admissible circuits return `method = "piecewise_linear_segment"`. Behavioral / unsupported circuits return a typed `failure_reason` and `ok() == false`.
- [ ] 1.2 AD-driven Behavioral linearization — deferred to a follow-up. The pattern would seed the existing `assemble_residual` through `Eigen::AutoDiffScalar` to produce `J = ∂F/∂x` at the operating point. Today we bail out with `failure_reason = "linearize_non_admissible_behavioral_device"` so the AC-sweep / FRA paths can still land on PWL converters; mixed-mode circuits remain a known gap until 1.2 lands.
- [x] 1.3 [`test_frequency_analysis_phase1.cpp::Phase 1.3 RC`](../../../core/tests/test_frequency_analysis_phase1.cpp): linearization of an RC + V-source circuit places `+C` on `E[out, out]`, `-1/R` on `A[out, out]`, `+1/R` on `A[out, in]`, and `C = I` over the full state. RLC follow-up case solves `A·v = s·E·v` and confirms the underdamped pair `s = -α ± jω_d` (α = R/2L, ω_d = √(ω₀² − α²)) appears in the finite-eigenvalue set within 1e-3 relative tolerance.
- [x] 1.4 Failure reporting: `Phase 1.4` test pins that a circuit with a Behavioral-mode diode produces `ok() == false`, `method == "non_admissible"`, and `failure_reason` containing `"non_admissible"`. Degenerate-operating-point detection (rank deficiency) inherits from the DC OP solver's diagnostics — `linearize_around` does not re-validate beyond what the segment engine already gates.

## Phase 2: AC sweep core
- [x] 2.1 [`AcSweepOptions`](../../../core/include/pulsim/v1/simulation.hpp) (`f_start, f_stop, points_per_decade, num_points, scale: Logarithmic|Linear, perturbation_source, measurement_nodes, t_op, use_dc_op, x_op`) plus the per-measurement `AcMeasurement` (`node, state_index, magnitude_db, phase_deg, real_part, imag_part`) and the umbrella `AcSweepResult` (`success, failure_reason, frequencies, measurements, total_factorizations, total_solves, wall_seconds`).
- [x] 2.2 / 2.3 [`Simulator::run_ac_sweep`](../../../core/src/v1/simulation.cpp) flow: optional auto-DC-OP → PWL admissibility gate → `assemble_state_space(M, N, _)` at `t_op` → build B-column from the named perturbation source (voltage source: unit on its branch row; current source: ±1 on its node rows; promoted `Circuit::find_connection`/`find_device` to public for the type dispatch) → log/linear frequency grid → for each ω, build `K(ω) = jω·M + N` as a complex sparse matrix and solve `K·X = B` via `Eigen::SparseLU<SparseMatrix<complex<Real>>>`. Per-frequency magnitude is clamped at -300 dB to keep `log10(0)` from producing `-inf` in the returned data; phase is unwrapped into `(-180, 180]`.
- [x] 2.4 Symbolic-factor reuse across the sweep: the sparsity pattern of `K(ω)` is the union of M and N's patterns and is constant across ω, so `solver.analyzePattern(K)` runs **once at sweep start**. Per-frequency cost is one `factorize` + one `solve`. This is the AC-sweep counterpart of the Phase-3 LRU cache from `refactor-linear-solver-cache`; the same architectural invariant (analyze-once, factor-per-frequency) underlies both.
- [x] 2.5 [`test_frequency_analysis_phase2.cpp`](../../../core/tests/test_frequency_analysis_phase2.cpp) — 27 assertions / 4 cases:
  - **RC low-pass** (R=1 kΩ, C=1 µF): -3 dB at f_corner = 1/(2πRC) within ±0.20 dB; -45° phase within ±2°; -40 dB / -90° asymptote at 100×f_corner; H(jω_c) ≈ (0.5 - 0.5j) within ±0.02.
  - **RL high-pass** (R=100 Ω, L=1 mH): same -3 dB / -45° corner, -40 dB / -90° asymptote.
  - **Series RLC** (R=10 Ω, L=1 mH, C=1 µF, ζ ≈ 0.05): peak magnitude at resonance matches 20·log10(Q) within ±1.5 dB; -90° phase at resonance; -80 dB / -180° asymptote two decades above.
  - **Failure mode**: unknown perturbation source → `success = false` and `failure_reason` contains `"not_found"`.

## Phase 3: FRA implementation
- [x] 3.1 [`FraOptions`](../../../core/include/pulsim/v1/simulation.hpp) (`f_start, f_stop, points_per_decade, scale, perturbation_source, perturbation_amplitude, perturbation_phase, measurement_nodes, n_cycles, discard_cycles, samples_per_cycle`) plus `FraMeasurement` (per-node Bode arrays) and `FraResult` (`success, failure_reason, frequencies, measurements, total_transient_steps, wall_seconds`).
- [x] 3.2 [`Circuit::set_ac_perturbation`](../../../core/include/pulsim/v1/runtime_circuit.hpp) — public API that overlays `ε·cos(2π·f·t + φ)` on a named source's RHS during `assemble_state_space`. Voltage sources receive the perturbation at their branch row; current sources receive `±ε·cos(...)` at their node rows. Cosine convention (not sine) so the input phasor is `ε·e^{jφ}` (real for φ=0), matching the AC sweep B-column convention so FRA's Goertzel output divided by ε is directly `H(jω)` — same magnitude/phase reference as `run_ac_sweep`.
- [x] 3.3 [`Simulator::run_fra`](../../../core/src/v1/simulation.cpp): per-frequency: configure the perturbation, force trapezoidal integrator + fixed `dt = period / samples_per_cycle`, run `n_cycles · samples_per_cycle` steps via `run_transient`, capture each measurement node via `SimulationCallback`, drop the first `discard_cycles · samples_per_cycle` warmup samples, mean-subtract the rest (removes DC operating-point offset), Goertzel single-bin DFT at the perturbation frequency. Apply a `e^{-j·ω·dt/2}` correction to the Goertzel result — the half-step compensates for the trapezoidal midpoint input timing (input is averaged at `t + dt/2`) cancelled against the after-step sample capture (output is observed at `t + dt`), netting `+ω·dt/2`. Returns Bode magnitude/phase per node.
- [ ] 3.4 Adaptive `n_cycles` convergence check — deferred. Today the user picks `n_cycles` / `discard_cycles` manually; the defaults (6 total, 2 discard) work cleanly on first-order RC linear circuits to within the 5° / 1 dB gate. Self-tuning the cycle count by re-running until the Goertzel result stabilizes is the natural next refinement.
- [x] 3.5 [`test_frequency_analysis_phase3.cpp`](../../../core/tests/test_frequency_analysis_phase3.cpp) — 31 assertions / 3 cases:
  - **RC FRA-vs-AC parity** (gate G.1): 9 frequency points across two decades around the corner; FRA magnitude agrees with AC sweep within ±1 dB and phase within ±5° at every point.
  - **DC-offset isolation**: FRA on a 5 V DC + RC at the corner reports -3 dB / -45° regardless of the DC operating point — small-signal linearization is correctly isolated from the bias.
  - **Failure mode**: unknown perturbation source returns `success = false` with `failure_reason` containing `"not_found"`. Open-loop buck / closed-loop buck FRA on Behavioral converters (full Phase 3.5 spec) is gated on Phase 1.2 (AD-driven Behavioral linearization) — landing now would mean injecting on the PWL segment-primary path, which is correct but covers only the Ideal half of the converter taxonomy.

## Phase 4: Multi-input transfer-function matrix
- [x] 4.1 New `AcSweepOptions::perturbation_sources` (vector of names) replaces the single `perturbation_source` for MIMO sweeps. The implementation builds a multi-column B (one column per source) and solves `K · X = B` where X has `n_inputs` columns; per-frequency cost stays at one factorize + one multi-column solve. Backward-compat: when `perturbation_sources` is empty, falls back to the single `perturbation_source` and the result shape is identical to Phase 2.

Order of `result.measurements` is `(output_slot, input_source)` — outer loop is the measurement node. State-space identification consumers can iterate by `(node, source)` pairs to reconstruct the full transfer function matrix.

## Phase 7: YAML schema for analysis section
- [x] 7.1 New top-level `analysis:` YAML array. Each entry is a map with a required `type:` discriminator (`ac` / `ac_sweep` / `fra`) plus an optional `name:` label and the per-type fields. The parser populates [`SimulationOptions::ac_sweeps`](../../../core/include/pulsim/v1/simulation.hpp) and `SimulationOptions::fra_sweeps` (new fields). The two vectors are exposed publicly so users iterate and call the existing `Simulator::run_ac_sweep` / `run_fra` methods. The frequency-analysis types were extracted into a dedicated [`pulsim/v1/frequency_analysis.hpp`](../../../core/include/pulsim/v1/frequency_analysis.hpp) so `simulation.hpp` can hold vectors of them without forward-declaration gymnastics.
- [x] 7.2 [`yaml_parser.cpp`](../../../core/src/v1/yaml_parser.cpp) dispatches on the normalized `type:` string: `"ac"|"ac_sweep"|"ac-sweep"` → `AcSweepOptions`, `"fra"` → `FraOptions`, anything else → typed error `"unknown analysis type"`. Per-type fields are parsed via the existing `parse_real` / `parse_int_scalar` / `parse_string_scalar` helpers; `scale: log|linear` is normalized via `normalize_key` so case and synonyms (`logarithmic`, `lin`) all resolve. The new fields are also populated on `AcSweepOptions::label` / `FraOptions::label` for telemetry use.
- [x] 7.3 Strict mode rejects: (a) unknown `type:` values (`PULSIM_YAML_E_PARAM_INVALID`), (b) unknown per-entry fields like `bogus_param` (`PULSIM_YAML_E_UNKNOWN_FIELD`), and (c) `analysis:` not being a sequence (`PULSIM_YAML_E_TYPE_MISMATCH`). Root-level `validate_keys` was extended to allow `analysis` alongside `simulation` / `components` / `models`.
- [x] 7.4 [`test_frequency_analysis_phase7_yaml.cpp`](../../../core/tests/test_frequency_analysis_phase7_yaml.cpp) — 50 assertions / 6 cases:
  - **Round-trip parser**: RC YAML with one `ac` + one `fra` populates both vectors with all fields agreeing with the YAML inputs (frequencies, scale, source, measurement nodes, n_cycles, samples_per_cycle, label).
  - **End-to-end**: parse YAML → construct Simulator → `run_ac_sweep(opts.ac_sweeps[0])` → reproduce the analytical RC corner (-3.01 dB / -45° within ±0.20 dB / ±2°).
  - **Strict — unknown type**: `type: bogus` produces `"unknown analysis type"`.
  - **Strict — unknown per-entry field**: `bogus_param: 42` inside an AC entry produces `PULSIM_YAML_E_UNKNOWN_FIELD`.
  - **Multiple analyses**: two AC sweeps with distinct labels (`low_band`, `high_band`) parse in YAML order, each runs successfully against the shared `Simulator` (reuses DC OP automatically — `dc_operating_point` is idempotent for a fixed circuit).
